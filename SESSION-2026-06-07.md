**Feedback:** PCA 2D projection loses massive information. t-SNE or UMAP would give tighter visual clusters.

**Done:** Replaced `pca_2d()` with `umap_2d()` using `umap.UMAP(n_components=2, metric='cosine', n_neighbors=15, min_dist=0.1, random_state=42)`. Falls back to PCA if `umap-learn` is not installed.

**Rationale for UMAP over t-SNE:**
- UMAP preserves global structure better than t-SNE
- Faster to compute (3× ~2 min on this hardware vs t-SNE's ~10 min)
- Deterministic with random_state
- Cosine metric is appropriate for L2-normalized embeddings

**Cost:** Adds `umap-learn` dependency (with scikit-learn, numba, etc.). Full run takes ~3–5 minutes vs ~30s for PCA. Documented in GUIDE.md.

## Architectural Decisions

### Why UMAP over t-SNE?
- **Global structure preservation**: UMAP better preserves the overall geometry of the 300-D space. t-SNE tends to scatter clusters irrespective of inter-cluster distances.
- **Performance**: UMAP ~2 min/model, t-SNE ~8–10 min for 1,790 points at this perplexity range.
- **Determinism**: UMAP with `random_state=42` gives reproducible results.
- **Cosine metric**: Both support it, but UMAP's neighbor graph construction benefits from the same cosine metric used for the neighbours tab.

### Why centroid-based cuisine directions over paper's full probe set?
The paper provides 27 continuous probe labels + 8 cuisine tags, but the supplementary CSV bundle only contains the raw embedding vectors — not the probe score vectors or per-ingredient cuisine tags. To reconstruct the paper's exact cuisine directions we'd need the Claude-tagged cuisine labels per ingredient.

**Our approach:** The Core model's GMM mode atlas (194 emergent clusters, coherence 0.833) provides the closest approximation. For each of the 8 cuisine macro-regions, we collect members of modes whose labels explicitly name that cuisine (e.g., "East Asian fermented sauces", "Italian-Mediterranean deli", "Latin American chiles"). The centroid of those members gives the direction vector. For cuisines with weak mode coverage (Japanese, South Asian), we supplement with curated keyword seeds.

→ See `tools/compute_cuisine_directions.py` for the full mode-to-cuisine mapping.

### Why split into 4 JSONs instead of streaming or lazy chunks?
- **Simplicity**: The app has zero build step and zero dependencies. Lazy-loading 3 static files is browser-native `fetch()`.
- **Granularity**: Model-level granularity (3 files) is right-sized — loading individual neighbours/vectors at finer granularity would change the data format significantly.
- **Cache-friendly**: Each model JSON is ~4 MB — comfortably under browser cache limits, and separate URLs means a revisiting user re-downloads only what they interact with.

### Why keep base64 float32 packing?
- ~2.5:1 compression vs raw CSV text floats (~4 MB JSON vs ~11 MB CSV-converted)
- Float32Array decoding is fast (~100ms for 4 MB on modern hardware)
- JSON is compressible further at the HTTP layer (gzip gives ~30% additional savings on binary data)
- Browser-native `atob()` + `Uint8Array` + `Float32Array` requires no dependencies

## Remaining Gaps (addressed)

### ~~1. Heuristic cuisine directions (not paper-exact)~~ **FIXED**

Cuisine direction vectors are now computed with **embedding-space nearest-neighbour expansion**:

1. Start with curated keyword seeds (25–40 ingredients per cuisine)
2. Compute seed centroid in Cooc's embedding space
3. Find all ingredients with cosine similarity ≥ 0.4 to that centroid
4. Expand the set to include them, recompute centroid

This propagates cuisine labels through the embedding space, recovering ingredients that are cuisine-consistent but weren't in the keyword list. For example, `east_asian` starts with ~37 seed keywords and typically expands to 80–120 ingredients. The threshold (0.4) was chosen manually — aggressive enough to add meaningful coverage, conservative enough to avoid cross-cuisine bleed.

Sensory directions (sweet, spicy, etc.) are unchanged — the keywords are unambiguous and don't benefit from expansion.

### ~~2. No cross-mode ingredient search~~ **FIXED**

Added a **"Search ingredient across all modes"** field in the Modes tab (below the existing filters). Type any ingredient name to see every mode (across the current model) that contains it. The matched ingredient name is highlighted in purple in each mode's member list. Works independently of the selected ingredient — you don't need to click an ingredient first.

Internally: `renderCrossModeSearch(query)` scans all modes in the current model's atlas, filters by partial match on member names, sorts results by match count descending, and renders with highlighted spans.
